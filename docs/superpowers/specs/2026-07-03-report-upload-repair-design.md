# 检查报告上传修复与补全设计

## 文档信息

- 日期: 2026-07-03
- 状态: 已确认方案，待实现
- 范围: 修复检查报告上传/查看/更新/删除可用性问题，并补齐记录表单页上传入口

## 背景

当前检查报告功能存在两类问题：

1. 已有附件流程存在可用性缺陷，导致上传、预览和详情页刷新不可靠或直接不可用。
2. 第二阶段设计文档承诺了记录表单页上传入口，但当前 `RecordForm.vue` 尚未提供该能力。

用户已确认新建记录页的目标交互为：

- 保存前可以先选择图片
- 点击保存记录后，先创建记录
- 创建成功后自动继续上传这些图片

## 目标

本次修复后应满足：

1. 记录详情页中的检查报告可正常上传、查看、编辑分类/标签、删除。
2. 新建记录页可在保存前选择图片，并在记录创建成功后自动上传。
3. 编辑记录页可直接上传图片。
4. 上传和图片预览地址与现有前后端环境变量配置一致。
5. 上传、删除、编辑附件后，详情页和表单页状态能正确刷新。
6. 后端避免对无效 `recordId` 先落盘再返回 404，减少孤儿文件。
7. 为关键行为补充最小回归测试。

## 不在本次范围

1. 新增草稿附件表或临时文件绑定机制
2. 云存储迁移
3. 图片压缩、懒加载、批量排序
4. 重新设计附件分类体系

## 方案选择

采用前端“待上传队列”方案：

1. 在 `RecordForm.vue` 中维护待上传文件列表和每个文件的元数据。
2. 新建记录时，先提交记录表单。
3. 记录创建成功后，拿到 `recordId`，再调用现有附件上传接口上传已选文件。
4. 上传完成后跳转到详情页，并由详情页拉取最新记录。

选择该方案的原因：

1. 交互符合用户要求。
2. 不需要引入后端草稿态，改动集中、风险较低。
3. 可以复用现有附件接口和权限模型。

## 设计

### 1. 前端上传地址与图片地址修复

统一规则：

1. `VITE_API_BASE_URL` 继续表示 API 根地址，如 `http://localhost:3000/api`
2. 附件上传使用 `${VITE_API_BASE_URL}/records/:recordId/attachments`
3. 图片访问基于 API 根地址反推站点根地址，最终访问 `/uploads/...`

这样可避免当前 `/api/api/...` 和 `/api/uploads/...` 的错误拼接。

### 2. RecordDetail 刷新逻辑修复

`RecordDetail.vue` 统一只保留一个刷新函数，例如 `refreshRecord`：

1. 首次加载调用它
2. 上传成功后调用它
3. 附件更新后调用它
4. 附件删除后调用它

避免当前模板和脚本引用了不存在的 `fetchRecord`。

### 3. RecordForm 增加待上传队列

新增一个表单内的检查报告区域，支持：

1. 保存前选择多张图片
2. 对每张待上传图片设置分类
3. 对每张待上传图片设置标签
4. 删除尚未上传的待上传项
5. 在编辑模式下展示已有附件画廊，并允许继续上传

交互分为两种模式：

1. 新建模式
   - 用户先填写记录并选择图片
   - 提交记录成功后，前端自动上传待上传队列
   - 上传完成后跳转详情页

2. 编辑模式
   - 页面已有 `recordId`
   - 可直接上传新图片
   - 可查看并维护已有附件

### 4. 上传组件职责调整

将 `AttachmentUpload.vue` 从“立即直传组件”扩展为可复用的两种工作模式：

1. `queue` 模式
   - 不直接发请求
   - 只负责选图、校验、编辑分类和标签
   - 把结果回传给父组件维护

2. `upload` 模式
   - 需要 `recordId`
   - 直接调用上传接口
   - 适用于记录详情页和编辑记录页

这样可以避免为新建页和详情页维护两套完全不同的 UI。

### 5. 上传执行策略

新建记录成功后，前端按单文件顺序上传待上传队列。

原因：

1. 更容易把分类和标签与对应文件绑定
2. 失败定位更清晰
3. 进度展示更简单

上传失败时：

1. 记录创建仍视为成功
2. 页面给出明确提示，说明“记录已创建，但部分或全部图片上传失败”
3. 成功上传的文件保留
4. 用户仍可在详情页继续补传

### 6. 后端防止无效 recordId 落盘

当前问题是 multer 先写文件，控制器后校验记录是否存在。

本次改为：

1. 在附件上传路由中，在 `upload.array(...)` 之前增加一个轻量中间件
2. 该中间件基于 `recordId + familyId` 查询记录是否存在
3. 若不存在，直接返回 404，不进入 multer
4. 若存在，再执行 multer 落盘和后续上传逻辑

这样可以在不重构上传架构的前提下阻止明显的孤儿文件产生。

### 7. 测试策略

新增最小回归测试，覆盖：

1. 前端:
   - 附件 URL 构造
   - 新建记录时待上传队列在记录创建后触发上传
   - 详情页刷新函数存在并在事件后调用

2. 后端:
   - 无效 `recordId` 时，上传前校验中间件返回 404
   - 有效 `recordId` 时，请求可继续进入上传中间件

本次不引入完整 E2E，仅补足会直接回归的关键路径。

## 风险与处理

1. 风险: 新建记录成功但部分图片上传失败
   - 处理: 明确区分“记录创建成功”和“附件上传结果”，并提示用户可在详情页继续处理

2. 风险: 复用上传组件时职责变复杂
   - 处理: 明确区分 `queue` 与 `upload` 两种模式，避免在一个分支里混合状态

3. 风险: 表单页状态变多
   - 处理: 将待上传项转换、上传执行、错误汇总拆成小函数，避免把所有逻辑塞进提交函数

3. 记录详情页可正常查看、更新分类/标签、删除图片，并正确刷新。
